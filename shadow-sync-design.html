How to read this. Three kinds of label: R1–R5 requirements (must-haves), C1–C7 design choices (real alternatives existed; we picked one with tradeoffs — C1–C4 are the major ones, C5–C7 are minor), and M1–M6 mechanisms (forced by a choice; no real alternative once the choice is made). 1. Requirements

What the product has to do. Five items, nothing more. Everything else in this document derives from these.

R1 One deployment supports multiple independent pairs of repos

A pair is two repos that share a subdirectory of content; the same deployment needs to run several pairs side by side — e.g. a backend pair and a frontend pair, each with its own two repos, its own dir, and its own .shadowignore. Pairs are named and independently addressable. Each pair is self-contained: one pair’s history, trailers, and shadow branches must not collide with another’s.

R2 Commits are copied with the same content and behaviour

A replayed commit has to look and act like the original from the target side: same tree, same author and author date, same merge topology, same handling of binary files, LFS pointers, symlinks, and submodules. git blame and git log --graph on the target must reflect what actually happened on the source.

R3 Each side can exclude files via .shadowignore

Each side controls what it exports. The filter is part of the repo’s own content, not a separately-deployed config.

R4 Only a declared subdirectory per side is in scope

A monorepo might share only backend/; the counterpart repo has that content at the root. The prefix is part of the config, not fixed in code, and the two sides don’t have to use the same prefix.

R5 Runs unattended on CI (GitHub Actions or equivalent)

No human is there to clear a prompt, resolve a conflict, or fix a line-ending surprise. The tool must be safe to retry, safe to schedule, and safe to run cross-platform (Linux and Windows runners).

2. Design choices

Points where we had a real alternative and picked one. Each card names the requirement(s) it derives from and the alternative it ruled out. Mechanisms that fall out of a choice without a meaningful alternative of their own live in §3, not here.

C1 Land changes via plain git merge of a shadow/ branch

Derives from R2 · R5

Rejected alternative: force-push replayed commits onto the target’s main; auto-merge via a bot; or require consumers to use a special merge invocation (-s subtree, git subtree merge --prefix=<dir>, git merge --allow-unrelated-histories).

Sync is bidirectional (R2), so both sides can have new commits. Force-pushing replayed commits onto the target’s main would destroy the target’s own commits — violating R2 applied to the target side. Conflict resolution at sync time has no human under R5.

Instead we materialise incoming history on a tracking branch named shadow/<pair>/<branch> and let the team that owns main land it. The landing command has to be plain git merge origin/shadow/… — anything more exotic (subtree strategies, unrelated-histories flags) becomes a tax on every consumer and a new way to get it wrong. That commitment forces two more decisions downstream: how to bootstrap ancestry (C4) and how shadow branches are laid out (M1).

C2 Deduplicate replayed commits via git trailers, scoped by source remote

Derives from R1 · R5

Rejected alternative: git notes (separate ref, easy to drop on a fresh clone); a sidecar mapping file outside git; or content-based dedup on tree SHAs (silently drops legitimate no-op commits).

CI reruns, manual retries, and cron-triggered syncs all have to converge to the same result under R5. Each replayed commit carries Shadow-replayed-<sourceRemote>: <sha>; before replay, we scan the target’s shadow branches for that pair and build a source SHA → target SHA map. Anything already present is skipped. Scoping the trailer by source remote keeps R1 honest: two pairs that both read from origin (a backend pair and a frontend pair in workspace mode, for example) can’t step on each other’s trailers.

Keying dedup on source SHAs is what forces per-commit replay as a mechanism (M4) — a whole-repo filter-repo pre-pass would mangle those SHAs before we got a chance to put them in trailers.

C3 Support workspace mode and orchestrator mode from the same codebase

Derives from R5

Rejected alternative: commit to one mode only — either the tool must live inside one of the synced repos, or it must run from a standalone orchestrator.

R5 doesn’t pick where the CI job lives. Workspace mode (tool runs inside one of the synced repos, that side is origin) is the simplest setup; orchestrator mode (tool runs from a third, standalone repo; both sides are external) is the cleanest separation. The engine takes both because the choice belongs to the consumer, not to us.

Supporting both is what forces the CI mechanisms in §3: auth has to handle both “caller is one of the synced repos” and “caller is a third repo” (M5), and CI logic has to be callable from whichever repo the consumer puts it in (M6).

C4 Bootstrap by seeding at the divergence point, not by replaying each source’s full history

Derives from C1

Rejected alternative: sync from init — replay every commit from each source repo into the target in chronological order, preserving all history, and merge the two full histories together on the target side.

C1 demands a merge-base between source and target. Two real ways to establish one:

  • Seed at divergence (what we do): create a single commit anchoring the two trees at the moment sync begins. Source history before that point stays on the source repo; the target sees a fresh divergence point and goes from there.
  • Sync from init: replay every commit from each source repo into the target in chronological order, then merge the two histories. The target becomes the union of both sources’ full histories plus its own.

Sync-from-init gives deeper archaeology on the target (git blame reaches back forever), but it’s slow to bootstrap (potentially millions of commits), rewrites the target’s own history to include two previously-independent histories, and confuses chronology whenever a source was initialised before the target was. Seed-at-divergence is fast and reversible; the cost is that pre-seed git blame on the target resolves to the seed commit and has to be chased back to the source repo.

Implementation: shadow-setup creates one commit per pair+branch via commit-tree, reusing the target’s current tree and parenting the target’s tip. A Shadow-seed trailer points at the source tip. Tree unchanged → no content drift. Target tip as parent → ancestry established. The setup is idempotent under R5.

Degree of freedom within “seed at divergence”: a content-preserving seed (reuse the target’s tree, just add a trailer) vs. a content-modifying seed (overwrite the target’s tree with the source’s). The latter would trash the target’s files on main at seed time, so wasn’t viable.

Minor choices

Three more decisions with genuine alternatives but narrower blast radius than C1–C4. Listed briefly.

C5 Symmetric a / b endpoints; direction is a runtime flag

Derives from R1 · R2. Rejected: separate push/pull scripts, or a primary/secondary model where one side is source-of-truth.

One code path, tested once, serves both directions; push and pull can’t drift out of sync. Direction is chosen at the command line with --from a or --from b.

C6 .shadowignore is read from the source commit’s tree, not the current checkout

Derives from R3 · R5. Rejected: read from the current checkout at sync time; or from a single pinned ref (e.g. always main).

Every replay reads the ignore rules exactly as they were at that commit — deterministic under retry. Updating the filter is just a normal commit on each side, not a config redeploy.

C7 Pin cross-platform git behaviour via -c overrides on every git call

Derives from R2 · R5. Rejected: trust the user’s global git config; git config --local (mutates the repo we’ve been handed); or GIT_CONFIG_COUNT/_KEY_i env vars.

Every invocation prepends -c core.autocrlf=false -c core.safecrlf=false -c core.precomposeunicode=true -c core.quotepath=false. Linux and Windows runners produce byte-identical trees; the user’s global config can’t corrupt a replay.

3. Mechanisms — what the choices force us to build

Once a choice is made, some machinery follows without a real alternative. Six mechanisms. Each is forced by a choice (or by a combination of choices and requirements). Within a mechanism there is sometimes a narrow sub-choice — noted as a degree of freedom where it matters.

M1 Shadow branch tips carry the target’s full layout

Forced by C1

git merge on the target applies the shadow branch’s tree against the target’s own. If the shadow tip held only the synced dir, merging would need a subtree-style invocation to drop the files at the right prefix — which C1 rules out. So we compose the shadow tip’s tree to mirror the target’s layout: the synced dir carries the incoming changes, and everything outside dir is spliced through from the target’s current tip. Merge mechanics stay on our side; consumers keep running git merge origin/shadow/….

M2 Unmapped merge parents fall back to the target’s branch tip

Forced by R2 × C4

R2 demands that merge topology survive. C4 creates a hard boundary — anything before the seed doesn’t exist in the target. Both together guarantee some source merges will reference parents we can’t map (commits before the seed, or on branches we don’t sync). Wiring those parents to the target’s branch tip keeps the merge a merge and keeps git merge-base returning a real ancestor. The ancestry edge sometimes jumps further than it did on the source — see §4.

Degree of freedom: the fallback target. We pick target-tip. Dropping the parent (demotes the merge to a linear commit) and failing the sync (halts on legitimately out-of-scope history) are the alternatives we rejected.

M3 Build replayed commits via git commit-tree plumbing; never touch the working tree

Forced by R2 · R5

R2 requires explicit control over author name, email, author date, committer and parents — values we set via GIT_AUTHOR_*/GIT_COMMITTER_* on the commit-tree call. A porcelain checkout + commit approach would route through the user’s hooks, signing config, and working-tree state, any of which can corrupt the replay. R5 also means there may not be a working tree at all (orchestrator mode). Between these, plumbing isn’t a real choice — it’s the only path that survives both constraints.

M4 Process replay one commit at a time (not via a whole-repo pre-pass)

Forced by C2

C2 keys dedup on source SHAs — each replayed commit carries the original’s SHA in a trailer. A git filter-repo or git subtree pre-pass that produced a transformed “view” of each side would rewrite those SHAs before we ever got to the trailer stage, breaking dedup on every re-run. So we process commits one at a time: diff against the source parent, swap the prefix (R4), write the tree with M3, add the trailer. Renames and copies surface as delete+add pairs; we don’t depend on -M/-C heuristics.

M5 CI auth via insteadOf rewrites: caller-specific first, generic fallback

Forced by C3

C3 says the caller’s repo can be one of the synced pair or a third orchestrator. GitHub scopes GITHUB_TOKEN to the caller’s own repo; cross-repo operations need a PAT (EXTERNAL_REPO_TOKEN). The workflow installs two URL rewrites: a caller-specific one pointing the caller’s own URL at github.token, and a generic one pointing https://github.com/ at the PAT. Longest-prefix match wins, so a PAT scoped only to the B-side can’t accidentally break pushes to the caller’s origin.

Degree of freedom: the two-rule structure vs. a single wildcard rule. A single wildcard is simpler but fails whenever the PAT’s scope is narrower than the set of repos the workflow touches — it gets applied to URLs it has no access to and returns 403. Longest-prefix matching avoids that.

M6 Distribute CI logic via reusable workflow_call workflows

Forced by C3

Because C3 lets the tool live in different repos, consumers in different places all need to invoke the same sync logic. A reusable workflow_call workflow lets each consumer ship a thin caller that inherits updates from one central definition. Permissions are declared at the call site, so the caller can’t grant the callee more than it meant to. The copy-paste-YAML-per-consumer alternative drifts silently under R5.

4. Unexpected consequences

Ongoing properties of the design that weren’t the point of making any choice. Good, bad, and neutral.

+ unplanned benefit ! unplanned side-effect we live with

! Seed commits look odd in git log

From C4

Because seeds reuse the target’s tree and add only a trailer, they show up in history as no-content commits. First-time reviewers flag them as bugs. The diagram in shadow-sync-explained.html exists largely to explain that those nodes are load-bearing.

! Round-trip merges can make ancestry look jumpy

From M2

When a commit goes A → B, gets merged on B, and is synced back to A, the merge’s parent is re-wired through the dedup map so it lands on the original A commit. Topologically correct — merge-base still answers the right question — but visually the edge jumps across the graph. Our echo-mapping test exists specifically to lock this behaviour down.

! Commits that touch only non-pair paths are silently skipped

From M4

If a source commit changes only files outside the declared dir, the remapped tree is identical to its parent’s and the commit is not replayed. Correct by design, but users periodically ask “where did my merge commit go?” The answer is usually “it had no files in the pair’s dir”.

! Multi-pair configs force one PAT to cover every listed repo

From R1 × M5

The sharpness of this one is specifically an R1 consequence: as soon as a deployment has more than one pair, EXTERNAL_REPO_TOKEN (one secret per workflow) has to cover every external repo across every pair. Adding a new pair without widening the PAT silently fails at push time. There is no per-pair credential.

+ Running the same sync locally after CI “just works”

From C2 + M3

Trailers are the dedup key and the working tree is never touched, so a developer can fetch the target remote and re-run the sync from their workstation. The second run is a no-op. We didn’t plan for this — it’s a happy side-effect of idempotence plus plumbing-only replay.

+ One bug fix covers both directions

From C5

Symmetric a/b means push and pull are the same code. A regression found on one direction is fixed on both at once; a test that covers pull covers almost all of push. The test matrix stayed roughly half the size it would have been with an asymmetric design.

+ Retroactive .shadowignore changes work without ceremony

From C6

Because the filter is read out of the source commit’s tree, adding an entry to .shadowignore takes effect on the next sync without touching config, CI, or the other repo. Different branches can even carry different ignore rules — useful on release branches where the export surface is narrower.
