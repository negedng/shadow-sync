Shadow Sync — Design Plan

Requirements, the design choices we made, the mechanisms those choices forced on us, and the surprises that fell out.

How to read this. Three kinds of label: R1–R6 requirements (must-haves), C1–C8 design choices (real alternatives existed; we picked one with tradeoffs — C1–C5 are the major ones, C6–C8 are minor), and M1–M8 mechanisms (forced by a choice; no real alternative once the choice is made). 1. Requirements

What the product has to do. Six items, nothing more. Everything else in this document derives from these.

R1 One deployment, multiple sync pairs

A pair contains two repos that share a synchronized directory of content. The deployment can contain multiple pairs: one repo can have multiple subdirectories synced with different repos through pair mechanisms. For example backend and frontend are subdirectories on a monorepo, while they same context are in the root directory of a backend and a frontend repository.

R2 Commits are copied with the same content and behavior

A replayed commit has to look and act like the original from the target side: same tree, same author and author date, same merge topology. git blame and git log --graph on the target must reflect what actually happened on the source.

R3 Monorepo side can exclude files via .shadowignore

Each side controls what it exports. The filter is part of the repo’s own content, not a separately-deployed config.

R4 Runs unattended on CI (GitHub Actions or equivalent)

No human is there to clear a prompt, resolve a conflict, or fix a line-ending surprise. The tool must be safe to retry, safe to schedule, and safe to run cross-platform (Linux and Windows runners).

R5 No merge conflict on shadow branches

Shadow branch replays clean code from the other repo, it should not result in merge conflict.

R6 Multiple branches, multiple shadow branches

Each branch can have a shadow branch on the other side of the pair.

2. Design choices

Points where we had a real alternative and picked one. Each card names the requirement(s) it derives from and the alternative it ruled out. Mechanisms that fall out of a choice without a meaningful alternative of their own live in §3, not here.

C1 Each direction lands changes via a shadow/ branch on the target side

Derives from R2 · R4 · R6

Rejected alternatives: force-push replayed commits onto the target’s main; auto-merge via a bot; or commit to one direction (single shadow branch on a single side, treating the other side as source-of-truth).

Sync is bidirectional — both sides accumulate new commits between syncs, and R2 says neither side can be silently overwritten. The model that follows: each direction gets its own shadow branch on the target side, named shadow/<pair>/<branch>. The replay engine produces the branch; the team that owns the target’s main lands it via git merge when they’re ready.

C2 Shadow branch tip carries the target’s full outer tree, with only dir/ swapped in

Derives from C1 · R4 · R5

Rejected alternative: shadow branch tip contains only the rewritten synced content — nothing outside dir/.

Given C1 — we have a shadow branch on the target side — the next question is what its tip’s tree should look like. R5 sets the bar: when the consumer merges, the only conflicts they should see are the ones their own contributors caused inside dir/. Everything else — missing files, modified files, modify/delete pairs — is a phantom conflict the engine produced. Two valid shapes:

  • Mirror the outer layout (what we do): shadow tip’s tree equals the target’s current tree, with dir/ replaced by the replayed pair content. Plain git merge origin/shadow/… just works — the diff outside dir/ is empty by construction. Mirror layout forces a tree composition (M1). The shadow tip’s non-pair content has to be rewritten on every push to track the target’s current main, otherwise a later merge silently reverts any change the target made outside dir/ between syncs (root README edits, CI config, another pair’s directory).
  • Subtree-only: shadow tip’s tree contains only the rewritten synced content. Plain git merge would treat the absence of files outside dir/ as deletions and wipe most of the target’s tree, so consumers would have to run git subtree merge --prefix=<dir> or a scripted equivalent instead. Subtree-only forces a special consumer command. Forgetting it — running plain git merge instead — corrupts silently: most of the target’s tree gets wiped in one apparently successful merge. Under R4 there’s no human to spot the damage before it lands.

C3 Deduplicate replayed commits via git trailers, scoped by source remote

Derives from R1 · R4

Rejected alternative: git notes (separate ref, easy to drop on a fresh clone); a sidecar mapping file outside git; or content-based dedup on tree SHAs (silently drops legitimate no-op commits).

Syncs have to converge to the same result under R4. Each replayed commit carries Shadow-replayed-<sourceRemote>: <sha>; before replay, we scan the target’s shadow branches for that pair and build a source SHA → target SHA map. Anything already present is skipped. Scoping the trailer by source remote keeps R1 honest: two pairs that both read from origin (a backend pair and a frontend pair in workspace mode, for example) can’t step on each other’s trailers.

C4 Tool runs from a standalone orchestrator repo

Derives from R4

Rejected alternatives: workspace mode — the tool runs inside one of the synced repos, with that side as origin (no url needed); or support both from the same codebase, letting the consumer pick.

Both modes can satisfy R4 — the question is what the engine has to handle. Workspace mode looks attractive: one less repo to set up, one config field saved (the workspace endpoint omits url), and the consumer can run the sync directly from their checkout. The cost shows up in three places:

Supporting both from the same codebase nominally lets the consumer pick, but bakes workspace-mode complexity into every code path and roughly doubles the test matrix (every test has to exercise the workspace shape and the orchestrator shape). Picking one keeps the engine simple; we picked orchestrator.

The choice still forces M5 (CI auth, now simplified to a single insteadOf rule) and M6 (reusable workflow_call so the orchestrator’s thin caller workflow inherits engine updates).

C5 Bootstrap by replaying full source history, parented to target’s init

Derives from C1 · C2 · R5 · R6

Rejected alternatives: seed at divergence — create a marker commit on each target branch anchoring the two trees at the moment sync begins, and replay only commits after the seed; no anchor at all — skip merge-base establishment entirely and require git merge --allow-unrelated-histories on the consumer side.

C2 demands a merge-base between source and target so plain git merge (the contract from C1) finds one; R5 further demands that the merge produce no phantom conflicts when it does. Without a real merge-base, git falls back to either “unrelated histories” refusal or a 2-way merge that flags the entire other tree as additions — either way, R5 violation. Three ways to give git a real merge-base:

  • Sync from init (what we do): replay every commit from each source repo into the target. Unmapped source parents (roots, orphan-branch ancestors) anchor at the closest echo'd ancestor when one exists, otherwise at target’s init commit (M2) — so every shadow branch shares either the target’s init or a real round-trip point as a common ancestor with target/main. Source history is preserved on the target; git blame reaches back to the original commits.
  • Seed at divergence: create one no-op commit per pair+branch, anchoring shared ancestry without replaying prior history. Faster to bootstrap, but loses pre-seed archaeology and adds a separate setup phase consumers have to remember to run.
  • No anchor + --allow-unrelated-histories: don’t establish a merge-base at all. Replay produces orphan commits on shadow branches; the consumer passes --allow-unrelated-histories to git merge the first time each new shadow branch is merged. The engine simplifies (no init/echo anchoring for roots, no anchor scaffolding); the cost lands on the consumer. After the first merge, the merge commit relates the two histories — so subsequent merges of that branch don’t need the flag — but every new shadow branch’s first merge needs it again. Ruled out by C1: --allow-unrelated-histories is a special merge invocation, exactly the consumer-side tax C1 says the engine should absorb. For a single- branch deployment it’s a one-time hassle; in active multi-branch syncing it’s a recurring stumble.

Sync-from-init costs a one-time bootstrap proportional to source-side history (per-commit replay through commit-tree plumbing). For a fresh monorepo joining mature source repos this is run once locally and the resulting shadow branches pushed; subsequent CI syncs only handle deltas. The benefit is that the design has no separate setup step and no seed trailer to maintain — the empty init commit on the monorepo serves as the natural common ancestor for plain git merge from any pair’s shadow branch. Chronologies look jumpy (source commit dates predate the target’s init) but topology is correct and merge-base always answers.

Degree of freedom within “sync from init”: per-commit replay vs. a single bulk import (e.g. git filter-repo --to-subdirectory-filter followed by a merge). The bulk approach mangles SHAs before they can be put into trailers, breaking C3’s dedup — so per-commit replay is the only path consistent with the rest of the design.

Minor choices

Three more decisions with genuine alternatives but narrower blast radius than C1–C5. Listed briefly.

C6 Symmetric a / b endpoints; direction is a runtime flag

Derives from R1 · R2. Rejected: separate push/pull scripts, or a primary/secondary model where one side is source-of-truth.

One code path, tested once, serves both directions; push and pull can’t drift out of sync. Direction is chosen at the command line with --from a or --from b.

C7 .shadowignore is read from the source commit’s tree, not the current checkout

Derives from R3 · R4. Rejected: read from the current checkout at sync time; or from a single pinned ref (e.g. always main).

Every replay reads the ignore rules exactly as they were at that commit — deterministic under retry. Updating the filter on the monorepo side is just a normal commit, not a config redeploy. The engine reads .shadowignore from whichever side is exporting in a given direction; in practice R3 only mandates the monorepo-side filter, but the symmetric implementation costs nothing extra and keeps the code path uniform under C6.

C8 Pin cross-platform git behaviour via -c overrides on every git call

Derives from R2 · R4 · R5. Rejected: trust the user’s global git config; git config --local (mutates the repo we’ve been handed); or GIT_CONFIG_COUNT/_KEY_i env vars.

Every invocation prepends -c core.autocrlf=false -c core.safecrlf=false -c core.precomposeunicode=true -c core.quotepath=false. Linux and Windows runners produce byte-identical trees; the user’s global config can’t corrupt a replay. Without these, the same line written on a Windows runner would land as CRLF on shadow and LF on the consumer’s tree, producing exactly the kind of phantom conflict R5 forbids.

3. Mechanisms — what the choices force us to build

Once a choice is made, some machinery follows without a real alternative. Eight mechanisms. Each is forced by a choice (or by a combination of choices and requirements). Within a mechanism there is sometimes a narrow sub-choice — noted as a degree of freedom where it matters.

M1 Compose git tree for cross-repo merge commits

Forced by C2

A cross-repo merge is a source commit whose parents straddle both repos: one ordinary shadow-chain parent, plus one echo’d parent that originated as a target-side commit (carries a Shadow-replayed-{targetRemote} trailer pointing back to its target-side hash). Replaying it onto target’s shadow needs a tree composed from two sources, because no single source-side tree captures both halves correctly:

  • Outer files (everything outside target.dir/) come from the echo’d parent. The shadow branch only carries snapshots of target/main — it has no independent outer history — so the shadow chain’s outer is always behind-or-equal to target/main’s current state. The echo’d parent is a target-side commit, so its tree’s outer is authoritative for the most recent round-trip point.
  • target.dir/ subtree comes from the shadow-chain parent (the first mapped parent’s target.dir/), with the source merge commit’s own diff overlaid on top via buildRemappedTree. The source side already resolved any merge conflicts inside the synced subtree (R5: each conflict resolved at most once), so we accept that resolution verbatim. The work-branch contributions that round-tripped through source aren’t yet in the shadow branch’s history, so they correctly appear as part of this merge commit when it lands on target’s shadow. 
parentTree  = compose(echo’d^{tree}, target.dir, shadow_parent:target.dir/)
finalTree   = parentTree + sourceMerge.diff      (via buildRemappedTree)
commit-tree finalTree -p shadow_parent -p echo’d_target_commit

The replayed commit ends up with parents [shadow_chain_parent, echo’d_target_commit] — a real merge between the work branch and the shadow chain. git log -- target.dir/file on target/main therefore credits the source-side merge as the integration point: original work-branch commits remain ancestors via the merge edge, not duplicated reconstructions. Linear shadow commits (no echo’d parent) skip the composition and inherit parentTree from the previous mapped parent’s tree directly — outer state propagates forward until the next cross-repo merge updates it.

Alternative considered: real git merge instead of tree composition. Three flavors, all rejected:

  • Echo + git merge x: x already an ancestor → Already up-to-date, no commit, intermediate outer never updates. R5 violated.
  • Drop echo + git merge x: merge-base falls back to init; files modified by both b and x conflict; no human to resolve. Even on success, would re-resolve what M already resolved on Fornax — violating “each conflict resolved at most once.” R4 + R5 violated.
  • Merge-trick: git merge --no-commit -s ours x + read-tree --prefix=dir/ M^{tree} + outer fix-up (-s ours keeps b'’s tree wholesale; outer needs an extra step). Same parents, same tree, same SHA as the plumbing route — mechanically worse:

The end result is provably identical: both routes assemble a commit with the same parent list, the same tree (m1’s tree projected through the path remap), and therefore the same SHA. Nothing in this table is about output correctness — the choice is purely about how robustly we get there. Plumbing wins on every operational axis below, so we take it.

Identical commit through a more fragile route. M’s resolution was a human decision, not an algorithmic function of three trees — anything that replays M has to read M^{tree}. The only question is whether to do it via plumbing (tree composition) or porcelain (working-tree overwrite). Same information, two routes; we take the plumbing one.

Degree of freedom for the target.dir/ source: compose with the shadow-chain parent’s dir/, or use the echo’d commit’s dir/ raw. We compose with the shadow chain. Raw would clobber earlier source-side dir/ contributions with the echo’d commit’s view of the synced subdirectory.

M2 Unmapped parents anchor at the closest echo’d ancestor, or target’s init

Forced by R2 × C5

Some source commits reference parents we can’t map: roots of orphan branches, or merges pulling in branches we don’t sync. R2 demands those merges stay merges; C5 demands every shadow chain reach a common ancestor with target/main. The unmapped parent is replaced with the closest mapped ancestor in the source’s history — in practice, the most recent echo’d commit (a target-side commit that round-tripped through). When no echo exists in the parent’s ancestry, the fallback is target’s init commit. The ancestry edge sometimes jumps further than it did on the source — see §4.

Why init and not target/main’s current tip. target/main moves between syncs (consumer commits outside the pair, which never enter the replay), so anchoring orphans at the current tip would carry whatever tree the consumer happens to have today into the shadow chain’s outer state. At consumer-merge time the shadow chain’s outer would then look like a fresh modification, silently reverting the consumer’s recent non-pair changes. Init is stable, predates everything, and lets M1 handle outer composition at the actual cross-repo merge points where a real, authoritative outer is available via the echo.

Subtlety we ran into: with a path filter, git rev-list --parents silently rewrites merge parents whose ancestors are TREESAME at the path, dropping them from the output. That breaks parent-fallback (we’d never see the parent that needed fallback). The replay engine therefore selects commits via path-filtered rev-list but reads each commit’s true parents via a separate git log --no-walk pass, bypassing simplification.

Degree of freedom: the fallback target when no echo’d ancestor exists. Picking target/main’s current tip (rejected) requires a separate push-time recompose to correct the resulting outer staleness; init avoids the problem entirely. Dropping the parent (demotes the merge to a linear commit) and failing the sync (halts on legitimately out-of-scope history) are also rejected.

M3 Build replayed commits via git commit-tree plumbing; never touch the working tree

Forced by R2 · R4

R2 requires explicit control over author name, email, author date, committer and parents — values we set via GIT_AUTHOR_*/GIT_COMMITTER_* on the commit-tree call. A porcelain checkout + commit approach would route through the user’s hooks, signing config, and working-tree state, any of which can corrupt the replay. R4 also means there may not be a working tree at all (orchestrator mode). Between these, plumbing isn’t a real choice — it’s the only path that survives both constraints.

M4 Process replay one commit at a time (not via a whole-repo pre-pass)

Forced by C3

C3 keys dedup on source SHAs — each replayed commit carries the original’s SHA in a trailer. A git filter-repo or git subtree pre-pass that produced a transformed “view” of each side would rewrite those SHAs before we ever got to the trailer stage, breaking dedup on every re-run. So we process commits one at a time: diff against the source parent, swap the prefix (R1), write the tree with M3, add the trailer. Renames and copies surface as delete+add pairs; we don’t depend on -M/-C heuristics.

M5 CI auth via a single insteadOf rewrite to the PAT

Forced by C4

Under C4 the orchestrator pushes only to external repos — every push is cross-repo, so GITHUB_TOKEN (scoped to the caller’s own repo) is never sufficient. The workflow installs one URL rewrite: https://github.com/ → https://x-access-token:${PAT}@github.com/. The PAT (EXTERNAL_REPO_TOKEN) must have Contents: Read+Write on every external repo named in shadow-config.json — both the A-side and every B-side. No layering, no caller-specific exception rules, no fallback to github.token.

M6 Distribute CI logic via a reusable workflow_call workflow

Forced by C4

The orchestrator (C4) is a separate repo from this one. Its workflow has to invoke our sync logic without copy-pasting YAML — otherwise every orchestrator drifts on its own schedule. A reusable workflow_call workflow lets the orchestrator ship a thin caller that uses: the central definition; engine updates flow through automatically. Permissions are declared at the call site, so the caller can’t grant the callee more than it meant to.

M7 Preflight gate: halt only on conditions the engine can’t handle

Forced by R2 × R4

R2 says the replay must faithfully reproduce source history. R4 says no human is around to inspect. Together they rule out the “run anyway and hope” failure mode — a CI job that returns exit 0 with corrupted output is worse than one that halts. The rule we apply: if the engine can produce a faithful tree, no check; if it can’t, halt. No advisory warnings — they’re log noise nobody acts on. runPreflightChecks() runs against each source branch tip before replay starts:

  • Shallow clone — halts. rev-list on a truncated history silently elides commits past the shallow boundary, and findEchoAnchor’s ancestor walk truncates the same way. The replay would produce a tree-correct but ancestry-wrong shadow branch. The fix (git fetch --unshallow) is one command, so blocking is cheap and right.
  • Case conflict in tree — halts on Windows/macOS. With core.ignorecase=true (the default on those platforms), update-index --index-info folds two paths that differ only in case into one index entry, silently dropping the loser from the replayed tree. Linux preserves both, so the check is platform-gated.

Degree of freedom: what counts as “can’t handle.” Submodules, symlinks, and LFS pointers all round-trip through the engine as ordinary tree entries (gitlink, symlink blob, pointer blob respectively), so they’re left unchecked — even though the resulting tree may not be useful in every consumer scenario. We accept that surface for the silence: a flagged-but-passing build pollutes the log without changing behavior, and operators stop reading warnings that never matter.

M8 Per-branch sync; each source branch gets its own shadow ref

Forced by R6

R6 says each source branch gets its own shadow branch on the target. Mechanically: the engine enumerates source-side branches via git branch -r (filtering out shadow/* refs so we don’t recursively sync our own output), then runs the replay loop once per branch. Each push lands at shadow/<pair>/<branch> on the target, so syncing main and feature/x produces two distinct refs without sharing tip state.

A new source branch needs no per-branch bootstrap. Commits inherited from already-synced branches resolve through shaMapping (built from existing shadow refs at the start of every run); only the genuinely new commits get replayed. If the new branch’s earliest unmapped commit has no echo’d ancestor, M2 anchors it at target’s init — the same common ancestor every shadow branch shares.

Stale shadow detection: when syncing every branch (no --branch flag), the engine compares the set of remote-tracking shadow/<pair>/* refs against the source branches it just synced. Anything left over — a shadow ref whose source branch was deleted — gets logged with the explicit git push --delete command. The engine doesn’t auto-delete: deletion of a branch on the target is destructive and crosses out of R4’s “safe to schedule” comfort zone.

4. Unexpected consequences

Ongoing properties of the design that weren’t the point of making any choice. Good, bad, and neutral.

+ unplanned benefit ! unplanned side-effect we live with

! First sync is proportional to source history length

From C5

Because we replay every source commit per-pair on bootstrap, the first sync against a mature source repo can take minutes to tens of minutes. The expected workflow is to run the first sync locally, push the resulting shadow branches, and let CI pick up only the deltas afterward. Steady-state sync runs are unaffected — the Shadow-replayed-<remote> trailer makes already-replayed commits a no-op skip.

! Source commit dates predate target init

From C5

Replayed commits keep their original author dates, which are typically older than the target’s init commit. git log --date-order on the target therefore looks jumpy — old commits appearing “after” the target’s init. Topology is correct (merge-base answers properly) and git log --topo-order reads naturally; only date-ordered views surprise.

! Round-trip merges can make ancestry look jumpy

From M2

When a commit goes A → B, gets merged on B, and is synced back to A, the merge’s parent is re-wired through the dedup map so it lands on the original A commit. Topologically correct — merge-base still answers the right question — but visually the edge jumps across the graph. Our echo-mapping test exists specifically to lock this behaviour down.

Concrete walkthrough:

1. A commits `a`              A/main:    init_A —— a

2. Push A→B replays           B/shadow:    a′
                              a′ carries trailer:
                                  Shadow-replayed-A: sha(a)

3. B merges shadow,           B/main:    init_B —— m —— b
   then commits `b`                                  ╱
                              B/shadow:    a′ ———╱

4. Pull B→A replays           Replay sees m has parents [init_B, a′]:
                                  init_B  → M2 fallback (no echo’d ancestor; uses init_A)
                                  a′      → echo-mapped to `a`
                                              (original A commit, same SHA)

                              A/shadow:  m′ —— b′
                                          ↑
                                          ⌊ second parent of m′ IS `a`

5. A consumer merges shadow.

   git log --graph A/main:

       *  M           <-- consumer's merge of shadow
       |╲
       | *  b′
       | *  m′       <-- replayed B-side merge
       | |╲
       |/  |
       *   |  a        <-- m′'s second parent edge "jumps" back to `a`
       *   .  init_A

The edge from m′ back to a is what looks jumpy: m′ is a recent commit, a is on the main line further back, and the line between them crosses other commits in git log --graph output. Topologically it’s exactly right — git merge-base m′ A/main returns a correctly, no information is lost. The visual just looks unusual because most graph viewers aren’t used to merge edges spanning that distance.

! Commits that touch only non-pair paths are silently skipped (monorepo → external direction only)

From M4

When the source endpoint has a dir prefix (i.e., the source is the monorepo side), the engine uses git rev-list main -- dir/ to collect commits, so any commit that doesn’t touch a file under dir/ never enters the replay queue. Correct by design — a commit that edits only .github/workflows/ or another pair’s directory has nothing to send to the external repo — but users periodically ask “where did my merge commit go?” The answer is usually “it had no files in the pair’s dir.”

The other direction (external → monorepo, where the source has no prefix) replays every commit on the source branch — there’s no path filter to skip anything. So this asymmetry only bites monorepo developers wondering why a root-only commit didn’t propagate.

! Adding a new pair silently breaks if the PAT’s scope isn’t widened

From R1 × M5

With orchestrator mode (C4), one PAT covers every listed repo by design — that’s how M5’s single insteadOf rule works. The surprise is the failure mode when scope drifts out of sync with config: adding a pair to shadow-config.json without also widening the PAT’s repo allow-list silently fails at push time. The fetch succeeds (pull is read-only on the source); the push to the new target returns 403, the sync run is marked failed, and the operator goes hunting before realising the PAT was the missing piece. There is no per-pair credential to scope around it.

+ Running the same sync locally after CI “just works”

From C3 + M3

Trailers are the dedup key and the working tree is never touched, so a developer can fetch the target remote and re-run the sync from their workstation. The second run is a no-op. We didn’t plan for this — it’s a happy side-effect of idempotence plus plumbing-only replay.

+ One bug fix covers both directions

From C6

Symmetric a/b means push and pull are the same code. A regression found on one direction is fixed on both at once; a test that covers pull covers almost all of push. The test matrix stayed roughly half the size it would have been with an asymmetric design.

+ Retroactive .shadowignore changes work without ceremony

From C7

Because the filter is read out of the source commit’s tree, adding an entry to .shadowignore takes effect on the next sync without touching config, CI, or the other repo. Different branches can even carry different ignore rules — useful on release branches where the export surface is narrower.
