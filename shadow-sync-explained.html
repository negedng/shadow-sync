Shadow Sync

Technical reference — bi-directional git commit replay with path-prefix remapping

One-Pager Overview

Purpose

Shadow Sync keeps two git repositories in sync when one contains a subdirectory that mirrors the root of the other. Commits are replayed one-by-one — preserving authorship, timestamps, branch structure, and merge topology — while the path prefix is added or stripped as needed.

Each side receives a shadow/ branch that the team merges into their main branch when ready. Both teams keep full control over when changes land: no force-push, no rewritten history.

How the repos are related

RepoA (monorepo) RepoB (standalone) backend/src/app.ts <—— sync ——> src/app.ts backend/README.md <—— sync ——> README.md frontend/... (not synced) infra/... (not synced)

RepoA and RepoB are symmetric endpoints. Which one is "source" and which is "target" depends on the direction flag at runtime (--from a or --from b). The dir field in the config sets the path prefix per side ("backend" vs. "" for root).

The tool runs from a workspace — either one of the repos itself (the other is added as a remote) or a standalone orchestrator repo (both are added as remotes).

Walkthrough: Joining Two Mature Repos into a Monorepo

Concrete scenario. You already have org/frontend (four years of history, active PRs) and org/backend (three years of history, active PRs). You want a new org/monorepo that holds both under frontend/ and backend/, with every future change flowing between the monorepo and the original repos in both directions.

Starting and target state

BEFORE ═══════════════════════════════════════════════════════ org/frontend org/backend ├─ src/ ├─ src/ ├─ package.json ├─ go.mod └─ README.md └─ README.md (4 years of history, open PRs) (3 years of history, open PRs) org/monorepo (empty, or existing with unrelated root tooling) AFTER ═══════════════════════════════════════════════════════ org/monorepo ├─ frontend/ ←── sync ──→ org/frontend │ ├─ src/ │ ├─ package.json │ └─ README.md ├─ backend/ ←── sync ──→ org/backend │ ├─ src/ │ ├─ go.mod │ └─ README.md ├─ README.md (monorepo-only, not synced) └─ .github/workflows/ (monorepo-only, not synced)

Step 1 — Start with an empty (or near-empty) monorepo

No content copy is needed up front. The monorepo’s init commit (or whatever its current main tip is) becomes the natural common ancestor for both pairs’ shadow branches once the first sync runs. 

git clone https://github.com/org/monorepo.git
cd monorepo
# (or just git init + push if it doesn't exist yet)

Step 2 — Install and configure

Run in workspace mode: the monorepo is the workspace. a is origin with a dir prefix; b is the external repo at root. 

npm install negedng/shadow-sync cross-env tsx

Add the script to the monorepo’s package.json:

{
  "scripts": {
    "sync":  "cross-env SHADOW_CONFIG=./shadow-config.json tsx node_modules/shadow-sync/shadow-sync.ts"
  }
}

Create shadow-config.json:

{
  "pairs": [
    {
      "name": "frontend",
      "a": { "remote": "origin",        "dir": "frontend" },
      "b": { "remote": "frontend-repo", "url": "https://github.com/org/frontend.git", "dir": "" }
    },
    {
      "name": "backend",
      "a": { "remote": "origin",        "dir": "backend" },
      "b": { "remote": "backend-repo",  "url": "https://github.com/org/backend.git",  "dir": "" }
    }
  ]
}

Step 3 — First sync (bootstrap)

Replays each source repo’s full history into the matching shadow/ branch on the monorepo. Each pair’s root commit gets parented to the monorepo’s current main tip, so all shadow branches share the same common ancestor — plain git merge works from day one. 

npm run sync -- -r frontend --from b
npm run sync -- -r backend  --from b
git fetch origin
git merge origin/shadow/frontend/main
git merge origin/shadow/backend/main
git push origin main

The first sync is proportional to source-side history (per-commit replay). For mature repos this can take minutes to tens of minutes, so it’s typically run locally once and the resulting shadow branches pushed; CI then handles only deltas. The Shadow-replayed-<remote> trailer makes already-replayed commits a no-op skip on subsequent runs.

Step 4 — Sync, both directions

Frontend team commits to org/frontend (or merges a PR there):

# On the monorepo:
npm run sync -- --from b -r frontend
git fetch origin
git merge origin/shadow/frontend/main       # lands on monorepo/main

Monorepo contributor edits frontend/src/foo.ts:

# On the monorepo:
npm run sync -- --from a -r frontend
# On org/frontend (via normal git UI or CI):
git merge origin/shadow/frontend/main       # lands on frontend/main

Same pattern for backend with -r backend. In production you’ll wire up the two reusable workflows (§12) on a cron and let them run — the manual commands above are just for first-time testing.

What happens to open PRs on org/frontend?

They keep working exactly as before. When one merges into org/frontend/main, the next --from b -r frontend run picks the new commits up and replays them onto origin/shadow/frontend/main in the monorepo. The monorepo team then merges the shadow branch when they’re ready. If a PR and a monorepo-side change have edited the same lines, the conflict surfaces at the merge step — no different from a pre-monorepo world.

What about git blame for old code?

Replayed commits keep their original author, date, and message, so git blame frontend/src/foo.ts on the monorepo resolves to the actual commit that introduced each line, with the original SHA recorded in the Shadow-replayed-<remote> trailer. Deep archaeology works the same on both sides.

What if you want the reverse direction — split an existing monorepo out?

Same mechanics, flipped. Start with a monorepo, create two empty target repos (org/frontend, org/backend), and run the first sync with --from a. The monorepo’s history under each dir/ gets replayed into the new repos. Sync flows both ways from day one.

1. Configuration

A shadow-config.json file defines one or more pairs. Each pair connects two repo endpoints (a and b) with a stable name used in trailers and branch names. The file location can be overridden via the SHADOW_CONFIG environment variable; the default is shadow-config.json next to the tool. 

{
  "pairs": [
    {
      "name": "backend",
      "a": { "remote": "repo-a", "url": "https://github.com/org/repo-a.git", "dir": "backend" },
      "b": { "remote": "repo-b", "url": "https://github.com/org/repo-b.git", "dir": "" }
    }
  ],
  "trailers": { "replayed": "Shadow-replayed" },
  "gitConfigOverrides": {
    "core.autocrlf": "false",
    "core.safecrlf": "false",
    "core.precomposeunicode": "true",
    "core.quotepath": "false"
  },
  "maxBuffer": 52428800,
  "shadowBranchPrefix": "shadow"
}

Key fields

FieldPurpose
namePair identifier. Used in shadow branch names (shadow/backend/main).
remoteGit remote name. Use "origin" if the tool runs inside that repo.
urlRemote URL. Omit if the remote already exists (e.g. origin).
dirPath prefix on that side. "backend" means files live under backend/; "" means repo root.
trailersTrailer key base. replayed becomes Shadow-replayed-{remote} on every replayed commit.

2. Bootstrap from Init

There is no separate setup step. The first npm run sync against a fresh target replays each source branch’s full history into the matching shadow/ branch, parenting the very first replayed root commit to the target’s current main tip. That tip — typically the empty init commit on a brand-new monorepo — becomes the natural common ancestor for every shadow branch the tool ever creates, so plain git merge origin/shadow/… always finds a merge base.

Per-commit replay through plumbing. commit-tree reproduces each source commit on the target side with the same author, date, and message, plus a Shadow-replayed-<sourceRemote>: <originalSHA> trailer. A subsequent sync scans the target’s shadow branches for those trailers, treats any commit it finds there as “already replayed,” and skips it — making the operation idempotent and re-runnable from any environment.
Bootstrap cost. First sync is proportional to source-side history length. Mature repos can take minutes to tens of minutes (per-commit commit-tree is slower than a bulk import but preserves the trailer-based dedup that the rest of the design relies on). Steady-state syncs only handle deltas, since already-replayed commits are skipped via the trailer scan. Practical pattern: run the first sync locally once, push the resulting shadow branches, and let CI take over from there.
Unmapped parents anchor at the closest echo, or target’s init. Some source commits reference parents the replay can’t map — merges that pulled in branches we don’t sync, or roots of orphan branches merged into main. Each such parent is replaced by the closest already-mapped ancestor in the source’s history (typically a recent echo’d commit), or, if no echo exists, by target’s init commit. Merges stay merges, merge-base answers correctly, and the chain’s outer state is anchored at a real round-trip point rather than a snapshot of target/main’s current tip.

3. Steps of a Sync

One invocation of shadow-sync.ts runs the steps below. The same code handles both directions — "pull" (--from b) and "push" (--from a) — by swapping which endpoint is source vs. target.

  1. 1
    Resolve direction. Parse --from; determine source and target endpoints. validateName() rejects names containing .., leading -, or absolute paths (injection guard). Ensure both remotes exist (add or update URL if the config specifies one).
  2. 2
    Fetch both remotes. git fetch {source.remote} for the commits to replay, and git fetch {target.remote} so target-side objects — the targetInit commit (root of {target}/main) and the current shadow-branch tips used by the replayed-trailer scan — exist locally before the replay engine starts resolving them. On a fresh CI runner, neither is cached, so both fetches are mandatory up front.
  3. 3
    Determine branches. If --branch is given: just that one. Otherwise: all branches on the source remote, excluding shadow/ branches.
  4. 4
    Preflight checks. For each branch, scan the source tip for shallow clones and (on Windows/macOS) case-insensitive filename conflicts. Both halt the branch — the engine can’t produce a faithful shadow under either condition.
  5. 5
    Mirror history. mirrorHistory() in shadow-common.ts runs the replay engine (see next section).
  6. 6
    Reconcile and push. For each branch, compare replayedSHA with the target's current shadow/{pairName}/{branch} tip:
    • Identical, or replayedSHA is an ancestor of the current tip — skip (already up to date).
    • Current tip is an ancestor of replayedSHA — push fast-forward.
    • Divergent (neither is an ancestor) — build a reconciliation merge via git commit-tree with two parents (current tip and replayedSHA) using the replayed tree, then push that. See Merges.
    Note: even when replay produces no new commits, the branch may still advance — the echo mapping can point the source's tip at an existing target-side commit, which this comparison then fast-forwards to.
  7. 7
    Report stale branches. If syncing all branches, warn about shadow branches whose source branch no longer exists. Print the git push --delete command for manual cleanup.

4. The Replay Engine

mirrorHistory() in shadow-common.ts is the core of the tool. It composes six named helpers, each responsible for one step:

  1. 1
    Scan replayed-commit mapping. loadReplayedMappings() runs once per pair (not per branch). It collects every {target.remote}/shadow/{pair}/{branch} ref that exists and passes them all into a single git log ... --grep=^Shadow-replayed-{remote}:, then extracts each trailer into a Map<sourceHash, targetHash>. The map is pair-wide and shared across all branches in this sync, so a commit reachable from multiple branches appears exactly once — when the replay loop later processes it for the first branch, subsequent branches find it already mapped and reuse the existing replayed SHA instead of duplicating work.
  2. 2
    Collect commits in topological order. collectSourceCommits() runs git rev-list --topo-order --reverse from each branch tip, narrowing to commits that touch the source subdirectory (where dir is non-empty). Then a separate git log --no-walk --format=%H %P pass reads each commit’s true parent list. This second pass is what protects merge topology: rev-list --parents with a path filter silently rewrites away any merge parent whose ancestors are TREESAME at the path, which would otherwise demote merges to linear commits on the shadow side.
  3. 3
    Filter. filterNotReplayedCommits() drops commits already in the mapping (skipped — already replayed) and echo commits that carry the target's remote trailer (round-trip prevention — see Deduplication). For each echo, the original target-side hash is read out of the trailer and — if that commit is still reachable in the local repo — recorded in the SHA mapping as echo → original. Subsequent commits whose parent is the echo will then be wired to the original commit object directly.
  4. 4
    Resolve target init. targetInit is the root commit of {target}/main. Any source parent that doesn’t map (orphan-branch root, merge with an out-of-scope parent, or root commit at the start of bootstrap) is replaced by the closest mapped ancestor in the source’s history; if none exists, targetInit is used. Merges stay merges and the bootstrap chain anchors to the target’s init.
  5. 5
    Replay loop. replayCommits() walks the remaining commits in topo order: load .shadowignore from the source tree, compute the diff (see section 5), build a remapped tree (section 6), resolve parents via the SHA mapping, and create a new commit preserving original author, committer, and timestamps. Record the new SHA in the mapping for subsequent commits to reference.

5. How the Diff Works

The replay engine is diff-based, not snapshot-based. For each source commit, only files that changed are extracted and applied to the previous replayed tree. The result: each replayed commit shows exactly the files the source commit changed — no phantom additions or deletions from the surrounding tree.

Computing the diff

For a normal commit (has a parent):

git diff-tree -r {parent} {commit} -- {sourceDir}/

Output format: :oldmode newmode oldhash newhash status\tpath — every file that changed between the commit and its first parent, scoped to the source directory.

For a root commit (no parent):

git ls-tree -r {commit} -- {sourceDir}/

All files are treated as additions. The output is converted to diff-tree format with status A.

Supported status codes:

StatusMeaningAction
AAddedupdate-index --add --cacheinfo {mode},{hash},{path}
MModifiedupdate-index --add --cacheinfo {mode},{hash},{path}
DDeletedgit rm --cached -f {path}
RRenamedrm old path + update-index new path
CCopiedupdate-index new path (old stays)
TType changeRe-update-index at new mode

6. How the Tree Is Built

buildRemappedTree() constructs a new git tree for each replayed commit. It uses a temporary index file so the working directory is never touched.

  1. 1
    Load base tree into the temp index. If this commit has a mapped parent: git read-tree {parentTree} loads the parent's full tree. If this is the first commit: git read-tree --empty starts fresh.

    The base tree is the first mapped parent's tree — a full target-side tree. This is critical: the replayed commit contains all files in the target repo, not just the synced subdirectory. Without this, git merge would see every file outside the synced directory as deleted.
  2. 2
    Compute the diff as described in section 5.
  3. 3
    Remap paths and apply changes. For each changed file:
    • Strip source prefix: backend/src/app.ts → src/app.ts
    • Check .shadowignore: skip files matching any pattern
    • Add target prefix: src/app.ts → backend/src/app.ts (or leave as-is if target dir is root)
    • Update the temp index using the low-level action from the table above
  4. 4
    Write the tree. git write-tree produces a tree object from the temp index. That hash becomes the tree of the replayed commit.
Source commit (RepoB, dir=""): Replayed commit (RepoA, dir="backend"): diff: M src/app.ts base tree: full RepoA tree (from parent) A src/new.ts apply: M backend/src/app.ts D old-file.txt A backend/src/new.ts D backend/old-file.txt result: full RepoA tree with exactly 3 changes

Cross-repo merges and out-of-scope commits

The replay chain is path-scoped: trees inherit unchanged outside {pair.dir}/ from whatever the previous replayed commit had, which traces back through the chain to the M2-anchored root. That root is target’s init when no echo is around, or the most-recent echo when one exists. Either way, outer state inside the shadow chain represents a past point in target/main’s history, never the current tip.

Cross-repo merges — source merge commits with one echo’d target-side parent — are where the shadow chain’s outer gets refreshed. The composition fires inside the replay loop (composeCrossRepoMergeTree()): outer comes from the echo’d target-side parent’s tree (a real commit on target/main), and {target.dir}/ comes from the shadow-chain parent. The replayed commit lands as a proper merge between the work branch and the shadow chain, and the consumer’s git merge origin/shadow/{pair}/{branch} finds merge-base at that echo — far enough back that the shadow chain’s frozen outer matches the merge-base outer, so out-of-scope changes the consumer made since (root files, sibling pairs’ directories, CI config) appear as a one-sided modification on the consumer’s side and survive the merge unchanged.

This is what the “out-of-scope commits” (marked x and y in the flow diagram above) rely on. Those commits live outside {pair.dir}/ and the replay engine never touches them — the path filter drops them at commit collection, they never produce replayed counterparts, and yet they survive the round trip because the shadow chain’s outer is anchored far enough back to make those changes look like a consumer-side modification rather than a fresh shadow-side revert.

7. How Merges Work

Merging happens at three distinct levels:

7a. Merge topology preservation (during replay)

When source history contains merge commits, the replay engine preserves them. resolveTargetParents() maps each source parent hash to its replayed equivalent using the SHA mapping. Anything that doesn’t map (orphan merges, root commits at the start of a bootstrap, merge parents on branches we don’t sync) is replaced by the closest already-mapped ancestor — usually a recent echo’d commit — or by target’s init when no echo is reachable. Duplicates are removed. The replayed merge is created with all resolved parents: 

git commit-tree {tree} -p {parent1} -p {parent2} ... -m {message}

Because all branches are processed in a single topological pass, a commit reachable from multiple branches is replayed once and reused — preserving shared ancestry rather than duplicating commits.

Cross-repo shared ancestry via echo mapping: if a merge on the source brought in an echo commit (one that originated on the target), the filter pass records the echo's original target-side hash in the SHA mapping (see section 4, step 4). When the merge is replayed, that parent resolves to the literal original commit — not a re-replay and not the target's current branch tip. The replayed merge ends up with one parent on the source side (replayed normally) and one parent that is the genuine target-side commit object, shared by SHA. Later git merge on the target finds the real common ancestor and avoids duplicating work.

7b. Shadow branch reconciliation (during push)

After replay, the tool pushes the result to shadow/{pair}/{branch} on the target remote. There are three cases:

CaseConditionAction
Fast-forward Current shadow tip is an ancestor of the replayed SHA Push directly.
Up to date Replayed SHA is an ancestor of the current tip Skip — nothing to push.
Diverged Neither is an ancestor of the other Create a merge commit with both as parents. The replayed tree wins. Push the merge.

In steady state the sync tool is the only writer of the shadow branch, so divergence shouldn't happen — but the tool doesn't trust that assumption. Three realistic triggers:

  1. Two workspaces racing to push. A dev laptop and a CI runner (or two CI runners) sync at roughly the same time. Both compute a replay and both try to push. The replayed commits can have different committer identity or committer timestamp depending on which machine ran the tool, so they get different commit SHAs even though author and tree are identical. The first push wins; the second arrives to find its SHA is neither an ancestor nor a descendant of the shadow tip.
  2. Force-push / history rewrite on the source branch. If someone rewrites source history (rebase, reset --hard, filter-branch), the old source SHAs are gone and the trailers on the existing shadow branch now point at dead commits. On the next sync, none of the new source commits are in the "already replayed" mapping, so the engine replays them fresh rooted on targetInit (or, if any of the rewritten commits’ ancestors still happen to be mapped via an echo, on that echo). The new chain has no ancestor relationship to the existing shadow tip. Covered by test-pull-force-rewrite.ts phase 1.
  3. Manual intervention on the shadow branch. Against policy, but possible — a human force-pushes or commits directly to shadow/backend/main.

In all three cases, the right answer is "the replayed tree is the truth, but preserve both histories." Failing loudly would turn every concurrent sync into a paging incident; the reconciliation merge keeps the diverged tip reachable so no work is lost, while the tree matches what the source currently says.

The reconciliation merge:

// Take the replayed tree as the truth
syncedTree = git rev-parse {replayedSHA}^{tree}

// Build a merge commit with both parents but the replayed tree
pushSHA = git commit-tree {syncedTree} -p {currentSHA} -p {replayedSHA}
    -m "Merge replayed commits into shadow/backend/main"

7c. User-facing merge (after sync)

After the tool pushes shadow branches, the target team lands them using plain git: 

git fetch origin
git merge origin/shadow/backend/main

Because shadow commits carry the full repo tree (not just the synced subdirectory), git's three-way merge works correctly — files outside the synced directory are untouched, and only genuine edit conflicts need manual resolution.

8. Trailers & Deduplication

In a bidirectional system, replayed commits would bounce back and forth forever without deduplication. Shadow Sync solves this with git trailers — metadata lines appended to commit messages.

Trailer format

Shadow-replayed-repo-b: 3fcc1ed4a5c0bdc92f4a7...

Every replayed commit gets a trailer whose key embeds the source remote name (sanitized: non-alphanumerics → hyphens, collapsed, trimmed) and whose value is the original source commit hash. Because the remote name lives in the key, multiple sources coexist cleanly and echo-detection stays direction-specific.

Two layers of protection

  1. Skip already-replayed commits. Phase 1 of the engine (loadReplayedMappings()) scans the target's shadow branch history for this pair only and builds a sourceHash → targetSHA map. Commits already in the map are skipped this run. The scan is deliberately pair-scoped: the trailer key doesn't encode the pair name, so a broader --all scan would pick up trailers from other pairs that happen to share the same source remote and falsely flag a commit as already replayed. If the target shadow branch doesn't yet exist, the map starts empty (correct "no prior replays" state).
  2. Skip — and remap — "echo" commits. When syncing from A to B, the replayed commit on B carries Shadow-replayed-repo-a: …. Later, when the user merges that shadow commit into B's main and we sync B back to A, the engine sees the A-sourced trailer. Two things happen:
    1. The echo commit itself is dropped from the new-commits list (no replay — that would create a duplicate).
    2. The hash inside the trailer points back at the original A-side commit. If that commit is still reachable in the local repo, the engine records echo → original in the SHA mapping. Any later commit (notably a merge) whose parent is the echo will then resolve to the original commit object itself — same SHA, shared by both sides.
    The trailer key the engine looks for is direction-specific:
    • Syncing A→B: add Shadow-replayed-repo-a, skip anything carrying Shadow-replayed-repo-b.
    • Syncing B→A: add Shadow-replayed-repo-b, skip anything carrying Shadow-replayed-repo-a.
RepoA commit X (sha = abc123) | v sync --from a RepoB shadow commit X' (carries Shadow-replayed-repo-a: abc123) | v merge into RepoB main RepoB main now contains X', plus a merge commit M whose parents are [B/main_tip, X'] | v sync --from b X' is dropped (echo); mapping recorded: X' -> abc123 M is replayed as M' with parents [replayed-B-tip, abc123] ^^^^^^ literal A-side commit, shared by SHA
Trailer matching is line-anchored: an identifier that happens to appear in a commit body (e.g. "see Shadow-replayed-repo-a: abc1234 for context") is not treated as a real trailer — only proper trailer lines at the end of the message count. See test-push-content.ts, phase 5.

9. .shadowignore

Each repo can control which files it sends to the other side. Place a .shadowignore file at the root of the synced content:

RepoConfig dirFile location
RepoA"backend"backend/.shadowignore
RepoB"" (root).shadowignore

Patterns work like .gitignore basics:

  • * matches within a single path segment
  • ** matches across path segments
  • Lines starting with # are comments
CLAUDE.md
.cursor/
**/*.local
secrets/

The ignore file is auto-discovered per commit from the source tree (git show {commit}:{dir}/.shadowignore). Adding or removing patterns takes effect immediately on the next sync — no config change required.

10. Preflight Checks

Preflight only flags conditions the engine can’t handle. Anything it does handle — submodules and symlinks (replayed as their respective tree entries), Git LFS pointers (copied through as plain blobs) — is left unchecked. The remaining hazards both produce silently wrong output and so abort the branch:

CheckCodeReason
Shallow clone SHALLOW_CLONE rev-list on a truncated history silently elides commits past the shallow boundary. The replayed shadow looks fine but is missing ancestry.
Case-insensitive filename conflict CASE_CONFLICT With core.ignorecase=true (Windows/macOS default), update-index --index-info folds paths that differ only in case — one of Readme.md/readme.md is silently dropped from the replayed tree. Skipped on Linux.

Branches with preflight errors are skipped; the rest of the sync continues.

11. Programmatic API

The CLI entry point (shadow-sync.ts) also exports a programmatic API used by the test harness. You can run a sync in-process without shelling out: 

import { runSync, applyTestOverrides } from "shadow-sync/shadow-sync";

runSync accepts { pair?, from?, branch? } and returns { exitCode, stdout, stderr }. This is how shadow-tests/harness.ts runs 17 test files against ephemeral git environments.

Authentication: runSync shells out to git fetch and git push just like the CLI does — it doesn't handle credentials itself. Any push to a private remote (e.g. B-side repos on GitHub, or cross-repo pushes in orchestrator mode) still needs a Personal Access Token configured in the caller's environment, either via the remote URL (https://x-access-token:{PAT}@github.com/…), a credential helper, or an SSH key. In CI this is the same PAT the shadow-forward.yml workflow uses; in-process callers inherit whatever git auth the parent process has.
